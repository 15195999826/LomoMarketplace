# 冒烟测试 (Smoke Test)

## 使用场景
- 功能已经写完了，想快速验证是否正常工作
- 不想手动一个个测试，让 AI 自动跑
- 改了代码怕引入回归 bug

## 使用方式
```
/smoke-test                          # 测试最近改动的功能
/smoke-test 删除记忆按钮              # 测试指定功能
/smoke-test /api/memories 接口        # 测试指定 API
```

## 工作流程

### 第 1 步：分析已有实现

读取最近改动的代码（git diff 或指定范围），自动生成测试用例：

```
🔍 检测到以下功能点需要测试：
  1. DELETE /api/memories/:id → 应返回 200
  2. 删除后 GET /api/memories/:id → 应返回 404
  3. 删除按钮点击 → 应弹出确认框
  4. 确认删除 → 列表应自动刷新
```

同时选择合适的测试方式：
- API 端点 → curl / fetch 断言
- UI 交互 → Playwright 脚本
- 终端输出 → wsh 读取 + 内容匹配
- 纯函数 → 直接调用断言

### 第 2 步：自主测试循环（不要问我）

1. 编写测试脚本
2. 运行测试
3. 如果有失败：读错误输出 → 诊断 → 修代码 → 重跑
4. 重复直到全部通过（最多 8 轮）
5. 连续 3 轮卡在同一个测试 → 停下来问我

### 第 3 步：结果报告

```
🧪 冒烟测试报告
- 测试用例: 4 个
- 迭代轮数: 2 轮
- 第 1 轮: ❌ 测试 2 失败（删除后仍返回 200）→ 修复 delete handler
- 第 2 轮: ✅ 全部通过
- 构建检查: tsc --noEmit ✅
```

附修复的 diff 摘要（如果有修复的话）。
