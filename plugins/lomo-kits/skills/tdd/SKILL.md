# 测试驱动开发 (TDD)

## 使用场景
- 开始做新功能时，怕 Claude 理解错需求闷头写一堆然后全返工
- 想让 Claude 自己迭代修 bug，不用你一直盯着

## 使用方式
```
/tdd 实现一个删除记忆的按钮
/tdd 给 Worker 加崩溃自动重启功能
```

## 工作流程

### 阶段 1：设计测试用例（等我审批）

读取相关代码，列出验收标准（测试用例）：

```
✅ 测试 1: 点击删除按钮弹出确认框
✅ 测试 2: 确认后调用 DELETE /api/memories/:id，返回 200
✅ 测试 3: 删除成功后列表自动刷新，该条消失
✅ 测试 4: 点取消不触发删除
✅ 测试 5: 删除不存在的 ID 返回 404
```

同时说明测试方式：
- API 测试 → curl / fetch 断言
- UI 测试 → Playwright 脚本
- 终端测试 → wsh 读取输出 + 内容匹配
- 纯逻辑 → 单元测试

**等我说"可以"后再进入下一阶段。**

### 阶段 2：自主实现循环（不要问我）

1. 编写测试脚本
2. 编写功能代码
3. 运行测试
4. 如果有失败：读错误输出 → 诊断 → 修代码 → 重跑测试
5. 重复直到全部通过（最多 8 轮）
6. 连续 3 轮卡在同一个测试 → 停下来问我

### 阶段 3：结果报告

```
📊 TDD 报告
- 测试用例: 5 个
- 迭代轮数: 3 轮
- 第 1 轮: ❌ 测试 2 失败（API 路由未注册）→ 添加路由
- 第 2 轮: ❌ 测试 3 失败（前端没刷新列表）→ 加 refetch
- 第 3 轮: ✅ 全部通过
- 构建检查: tsc --noEmit ✅
```
